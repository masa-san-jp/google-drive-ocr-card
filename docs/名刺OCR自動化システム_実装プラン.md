# 名刺OCR自動化システム 実装プラン（導入計画書）

本ドキュメントは、「名刺OCR自動化システム_設計仕様書.md」の内容に基づき、システムを実際の業務環境へ導入・実装するための具体的な計画と手順をまとめたものです。

---

## 1. プロジェクトの目的とゴール

*   **目的**: Google Workspace標準機能（Google Drive内蔵OCR ＋ GAS）のみを用いて、名刺情報の自動データ化・名簿化システムを構築し、営業担当者および管理担当者の業務負荷を軽減する。
*   **ゴール**:
    *   追加費用ゼロ（外部API不使用）でのシステム稼働。
    *   スマートフォンからの画像アップロードだけで、自動的にスプレッドシートへデータが転記される状態の実現。
    *   社内の非エンジニアでも運用・保守が可能な体制の構築。

## 2. 導入ステップとスケジュール（全体像）

システム導入は以下の4つのフェーズに分けて進行します。全体の所要期間は約1〜2週間を想定しています。

| フェーズ | 概要 | 期間目安 |
| :--- | :--- | :--- |
| **Phase 1** | **環境準備**（Driveフォルダ・スプレッドシートの作成） | 1日 |
| **Phase 2** | **プロトタイプ開発とテスト**（GASの実装とOCR精度の検証） | 3〜5日 |
| **Phase 3** | **マニュアル作成と運用テスト**（一部メンバーでの試運用） | 2〜3日 |
| **Phase 4** | **本番稼働・社内展開**（全社または対象部門への展開） | 以降継続 |

---

## 3. 各フェーズの詳細タスク

### Phase 1: 環境準備（インフラ構築）

Google Workspace上で必要なファイルやフォルダの枠組みを準備します。

*   [ ] **1-1. Google Drive フォルダの作成**
    *   「名刺管理」親フォルダの作成
    *   サブフォルダ作成（`01_未処理`, `02_OCR中`, `03_処理済み`, `04_要確認`）
*   [ ] **1-2. スプレッドシート（名刺台帳）の作成**
    *   「名刺台帳シート」の作成（A〜L列のヘッダー設定）
    *   「OCRステータス（I列）」に対する条件付き書式（色分け）の設定
    *   「操作ログシート」の作成
*   [ ] **1-3. 権限（アクセス共有）の設定**
    *   関係者（テスト参加者・管理者のみ）への編集権限の付与

### Phase 2: プロトタイプ開発とテスト（GAS実装）

設計書に沿ってGASをコーディングし、Google Drive内蔵OCRを用いたデータ抽出の仕組みを構築します。

*   [ ] **2-1. GASプロジェクトの作成**
    *   スプレッドシートから「Apps Script」エディタの起動
    *   各フォルダIDおよびシートIDの定数設定
*   [ ] **2-2. APIの設定**
    *   GASの「サービス」から「Drive API (v3)」を追加
*   [ ] **2-3. スクリプトのコーディング**
    *   フォルダ監視とステータス変更（ファイル移動）機能の実装
    *   `Drive.Files.copy` を利用した画像ドキュメント変換（OCR実行）機能の実装
    *   正規表現を用いたテキスト抽出（パース）機能の実装
    *   スプレッドシートへの行追加およびログ書き込み機能の実装
*   [ ] **2-4. 単体テスト**
    *   さまざまなパターンの名刺画像（縦書き、横書き、手書き混在など）をアップロードし、OCRの抽出精度とエラーハンドリング（20文字未満の「要確認」行きなど）の動作を確認する。

### Phase 3: マニュアル作成と運用テスト

実際のユーザーにテストしてもらい、業務フローの使い勝手を確認します。

*   [ ] **3-1. ユーザー向けマニュアルの作成**
    *   スマートフォンのGoogle Driveアプリを使った名刺の撮影・アップロード手順書の作成
    *   スプレッドシートの「要確認」データの修正・確定手順書の作成
*   [ ] **3-2. 一部メンバーでのプレ運用（UAT）**
    *   2〜3名の営業担当者に実際に名刺をアップロードしてもらい、1時間ごとのトリガーで正しく処理されるか検証する。
*   [ ] **3-3. 定期実行トリガーの本設定**
    *   GASのトリガーを本番想定（例: 1時間おき、業務時間内）に設定する。
    *   エラー通知（実行失敗時のメール通知）の設定確認。

### Phase 4: 本番稼働・社内展開

*   [ ] **4-1. 社内へのシステム利用の周知**
    *   マニュアルの配布と利用開始のアナウンス
*   [ ] **4-2. 運用開始**
    *   管理者は最初の数日間、「操作ログ」と「04_要確認」フォルダを頻繁にモニタリングし、問題が発生していないか確認する。
*   [ ] **4-3. 改善対応（継続）**
    *   要確認データが多く発生する場合、抽出ロジック（正規表現）の微調整を行う。

## 4. リスクと対策（特記事項）

1.  **Google Drive標準OCRの精度限界**
    *   **リスク**: Gemini API等の強力なAIに比べ、背景が複雑な名刺やデザイン性の高い名刺では項目（会社名・氏名）がうまく抽出できず、「要確認」になる確率が高くなる。
    *   **対策**: あくまで「入力補助ツール」と位置づけ、担当者による最終的な目視確認と手作業での修正・補完を業務フローにおける必須ステップ（仕様）として現場に周知する。
2.  **GASの実行時間制限（6分ルールの壁）**
    *   **リスク**: 一度に大量の名刺（数十枚）がアップロードされた場合、画像→ドキュメント変換処理に時間がかかりタイムアウトする可能性がある。
    *   **対策**: スクリプト側で「実行開始から5分経過したらループを中断する」安全装置を設け、残りの未処理ファイルは次回のトリガー実行時に回す仕様にする。
3.  **スマートフォンカメラの保存フォーマット（HEIC問題）**
    *   **リスク**: 最新のiPhone等はデフォルトでHEIC形式で保存されるため、Google Driveの仕様上OCRがうまく作動しないケースがある。
    *   **対策**: マニュアルにて、スマホのカメラ設定を「互換性優先（JPEG）」に変更するよう促す。またはGAS側で検知してエラーログを出す。
