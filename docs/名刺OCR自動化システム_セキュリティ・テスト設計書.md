# 名刺OCR自動化システム セキュリティ・テスト設計書

本ドキュメントは、名刺OCR自動化システムにおいてAPIキーや個人情報（名刺データ）の漏洩を未然に防ぎ、安全な運用を実現するための実装方針と、それを担保するためのテスト設計を定義します。

---

## 1. セキュリティ実装要件（機密情報の保護策）

Google Apps Script (GAS) において、以下のルールを徹底して実装を行います。

### 1-1. APIキーのハードコーディング禁止
*   **方針**: GASのソースコード内（`Code.gs`等）に、Gemini APIキーやスプレッドシートIDなどの認証・アクセス情報を直接文字列として記述（ハードコーディング）することを**一切禁止**します。
*   **実装策**: GASの「**スクリプトプロパティ（PropertiesService）**」機能を利用します。
    *   APIキー等の機密情報は、GASエディタのUI（プロジェクトの設定 ＞ スクリプトプロパティ）から環境変数としてのみ登録します。
    *   コードからは `PropertiesService.getScriptProperties().getProperty('GEMINI_API_KEY')` のように動的に呼び出して使用します。
*   **効果**: ソースコードを他のユーザーに見られたり、誤って社内外にコードを共有・コピーしてしまった場合でも、APIキー自体は漏洩しません。

### 1-2. ログおよびエラーへの個人情報・機密情報の出力禁止
*   **方針**: 実行ログ（`console.log`等）およびエラー発生時の通知（メール等）に、APIキーや抽出した名刺の生データ（氏名、メールアドレス等の個人情報）を含めないようにします。
*   **実装策**:
    *   API呼び出しのエラーハンドリングにおいて、エラーレスポンスをそのままダンプせず、エラー原因（HTTPステータス等）とファイルIDなど個人が特定できない情報のみを記録します。
    *   APIキー変数をログ出力するデバッグコードは配置しません。

### 1-3. 権限とアクセス範囲の最小化
*   **方針**: Google Driveフォルダおよびスプレッドシートに対するアクセス権を「必要最小限」に設定します。
*   **実装策**:
    *   「リンクを知っている全員」によるアクセスは不可とします。
    *   システムを運用・利用する特定のGoogleアカウント（または特定のGoogleグループ）に対してのみ権限を付与します。
    *   GASの実行権限（`appsscript.json` の `oauthScopes`）も、必要最小限のスコープに絞ります。

## 2. セキュリティ・テスト計画

上記の実装が正しく行われていることを確認するため、以下のテストケースを実施します。

### テストセットA：APIキーと設定情報の秘匿性テスト

| テストID | テスト大項目 | テストケース（確認手順） | 期待される結果（合格条件） |
| :--- | :--- | :--- | :--- |
| SEC-A-01 | コードレビュー | `src/Code.gs` のソースコード全体を文字列検索し、「`AIza`」等のAPIキーらしい文字列や、スプレッドシートIDが直書きされていないか確認する。 | 該当する文字列が一切見つからないこと。定数はプロパティから取得する処理になっていること。 |
| SEC-A-02 | プロパティ取得テスト | GASのスクリプトプロパティからキーを削除（または間違った値に変更）してスクリプトを実行する。 | スクリプトがエラーとして異常終了し、後続の処理（未承認のAPI通信など）が行われないこと。 |
| SEC-A-03 | 権限分離テスト | システム管理者以外の一般アカウントでGASのソースコード閲覧画面にアクセスしようとする。 | GASエディタにアクセスできない（編集権限がない）こと。 |

### テストセットB：ログ・通知の安全性テスト

| テストID | テスト大項目 | テストケース（確認手順） | 期待される結果（合格条件） |
| :--- | :--- | :--- | :--- |
| SEC-B-01 | 正常系ロギング | テスト用の名刺画像（本人の名前など個人情報を含む）をアップロードし、スクリプトを正常実行する。その後、GASの「実行数（Execution）」画面からログを確認する。 | ログ上に「処理完了: 〇〇様の名刺.jpg」のような「ファイル名」は出ても、抽出されたJSONデータやメールアドレス文字列などが直接出力されていないこと。 |
| SEC-B-02 | 異常系ロギング（APIエラー） | 無効なAPIキーを設定した状態でわざとスクリプトを実行し、APIエラーを発生させる。エラーログを確認する。 | エラーログにAPI URLなどの通信生データが含まれず、APIキーがマスキングされているか、または全く出力されていないこと。（例：「API認証に失敗しました」等の安全なメッセージであること） |
| SEC-B-03 | 異常系ロギング（パースエラー） | 名刺画像ではないダミー画像（真っ黒な画像など）を送ってOCR解析不能状態を引き起こす。 | エラー通知やログに、ダミー画像に対するGeminiからの返答（生レスポンス内容）が不用意にダンプ出力されていないこと。 |

### テストセットC：インフラストラクチャー（権限）の安全性テスト

| テストID | テスト大項目 | テストケース（確認手順） | 期待される結果（合格条件） |
| :--- | :--- | :--- | :--- |
| SEC-C-01 | フォルダ権限確認 | ブラウザのシークレットウィンドウ（未ログイン状態）で、「名刺スキャン用フォルダ」や「名簿台帳スプレッドシート」のURLに直接アクセスする。 | アクセス権限がなく、ログイン画面またはエラー画面が表示されること。（リンクを知っていれば誰でも見られる状態になっていないこと） |

## 3. 次ステップにおける実装への落とし込み

本テスト設計をクリアするため、現在作成中のGASスクリプト（`src/Code.gs`）および初期セットアップ手順（`docs/setup_guide.md`）に対して以下の修正（リファクタリング）を行います。

1.  **`src/Code.gs` の修正**:
    *   ファイルの先頭に記述していた `const CONFIG = { GEMINI_API_KEY: "..." };` のようなハードコーディングを廃止。
    *   初期化処理等で `PropertiesService.getScriptProperties().getProperty("GEMINI_API_KEY")` を呼び出す形に書き換える。
    *   `console.log` 周りを見直し、エラーオブジェクトを丸ごと出力している部分を安全なメッセージに変更。
2.  **`docs/setup_guide.md` の修正**:
    *   ユーザー向けの設定手順に、「GASエディタの UI画面（歯車マーク）から スクリプトプロパティを設定する手順」を明確に追加する。
    *   共有設定（アクセス権限）の注意事項を強調表示する。
