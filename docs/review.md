## 1. プロジェクト概要

名刺写真をGoogle Driveにアップロードすると、GASの定期トリガーがGemini API（gemini-2.5-flash-lite）でOCR処理し、スプレッドシートに自動登録するシステム。

-----

## 2. 評価できる点

### 2.1 実用的な課題設定とゼロコストアーキテクチャ

Google Workspace（GAS + Drive + Sheets）+ Gemini API のみで完結する構成は、追加インフラ不要・ランニングコストほぼゼロで動く点が優れている。非エンジニアでもsetup_guide.mdを読めば導入できる設計は、ターゲットユーザーに合致している。

### 2.2 APIキーのセキュリティ管理

```js
const apiKey = properties.getProperty('GEMINI_API_KEY');
```

APIキーを`PropertiesService.getScriptProperties()`で取得し、CONFIGオブジェクトに直接記述しない方針が**コード上でも一貫して実装されている**。CLAUDE.mdでのコミット前`git diff`チェックの指示、CONFIGのプレースホルダー文字列との組み合わせでセキュリティ意識が明確。

### 2.3 PDF複数名刺への対応設計

PDFと画像でプロンプトを切り替え、PDFでは複数名刺の一括抽出に対応している。さらにレスポンスのパース時に「単一オブジェクトが返ってきた場合も配列に正規化する」防御的なコードが書かれている。

```js
if (Array.isArray(parsed)) return parsed;
if (parsed && typeof parsed === 'object' && !Array.isArray(parsed)) return [parsed];
return [];
```

Gemini APIが指示通りに配列で返さないケースへの後方互換対応は、LLM連携特有の不安定さを理解した上での実装として評価できる。

### 2.4 ファイル単位のtry-catch

1件のファイル処理失敗が他のファイルに波及しない設計は、バッチ処理の基本として適切。`while (files.hasNext())` ループ内で各ファイルをtry-catchで囲み、エラーログを出力した上で次のファイルに進む構造になっている。

### 2.5 エラーレスポンスのセキュリティ配慮

```js
const errorData = (() => {
  try { return JSON.parse(responseBody).error.message; } catch(e) { return "内容不明またはパースエラー"; }
})();
```

APIエラーのレスポンス全体をログにダンプせず、`.error.message`のみを抽出するIIFEは、APIキーやリクエスト内容がログに漏れるリスクを低減している。

### 2.6 generationConfig による JSON 強制

```js
"generationConfig": {
  "responseMimeType": "application/json"
}
```

Gemini APIの`responseMimeType`で出力形式をJSON強制しており、マークダウンコードブロック混入のリスクを構造的に低減。プロンプト側の指示（「バッククォートを含めないで」）と二重に防御している。

### 2.7 コメントの質

関数ごとにJSDoc風のコメントがあり、インラインコメントも「なぜそうしているか」を説明する意図コメントが中心。コードの可読性は高い。

## 3. 評価できない点・改善提案

### 3.1 【重大】設計仕様書とコードの乖離

設計仕様書では7つのモジュール（DriveService / GeminiService / ParseService / ValidationService / SheetsService / FileService / LogService）に分割する設計が書かれている。しかし実装は **`Code.gs` 1ファイル・関数3つ** のみ。

|設計仕様書の定義                               |実装状況                                    |
|---------------------------------------|----------------------------------------|
|`getUnprocessedFiles()`                |なし（`inputFolder.getFiles()`で直接取得）       |
|`getFileAsBase64()`                    |なし（`processBusinessCards`内にインライン）       |
|`parseGeminiResponse()`                |なし（`extractWithGemini`内にインライン）          |
|`normalizeDate()` / `normalizeAmount()`|なし                                      |
|`validate()`                           |なし                                      |
|`buildRowArray()`                      |なし（`sheet.appendRow`に直接配列リテラル）          |
|`buildFileName()`                      |なし（`processBusinessCards`内にインライン）       |
|`logInfo()` / `logError()`             |なし（`console.log` / `console.error`で直接出力）|
|ログシートへの記録                              |なし                                      |

**キャリアへの影響**: 仕様書と実装の乖離は「仕様書倒れ」と見なされる。面接で「この設計はなぜ実装に反映されていないのか」と問われるリスクがある。

**改善案**: GASでは複数`.gs`ファイルを作成できる。少なくとも`GeminiService.gs`と`FileService.gs`を分離し、設計と実装の対応関係を明示する。

### 3.2 【重大】エラーファイルの無限再処理

処理に失敗したファイルは`catch`でログ出力されるだけで、**入力フォルダに残り続ける**。次回トリガー実行で同じファイルが再処理され、同じエラーが永遠に繰り返される。

```js
catch (e) {
  console.error(`エラー発生ファイル: ${file.getName()}, エラー詳細: ${e.message}`);
}
```

壊れた画像ファイル、極端に大きいファイル、Gemini APIが処理できない形式のファイルが1つでもあると、毎回トリガーで無駄なAPI呼び出しが発生し続ける。

**改善案**: `03_エラー`フォルダを作成し、失敗ファイルを移動する。またはファイル名にプレフィックス`[ERROR]`を付与し、処理対象から除外するロジックを追加する。

### 3.3 【重大】テストが存在しない

テスト設計書の存在は確認できるが、テストコードがない。以下はロジックが純粋でテスト可能:

- `sanitizeFileName()` — 禁則文字除去の正規表現パターン
- JSONパース後の配列正規化ロジック（配列/オブジェクト/不正値のハンドリング）
- エラーメッセージ抽出のIIFE

**改善案**: clasp + Jest でローカルテスト環境を構築し、最低限パース系のユニットテストを書く。

### 3.4 【中】`sheet.appendRow()` のパフォーマンス問題

```js
extractedList.forEach((extractedData) => {
  sheet.appendRow([...]);  // 名刺1枚ごとにAPI呼び出し
});
```

PDFに10枚の名刺が含まれる場合、`appendRow`が10回呼ばれる。GASはSpreadsheet API呼び出し1回ごとにネットワークラウンドトリップが発生するため、大量処理時にGAS実行時間制限（6分）に到達するリスクがある。

**改善案**: `sheet.getRange(row, col, numRows, numCols).setValues(data)` で一括書き込みする。

### 3.5 【中】処理件数の上限がない

設計仕様書では`MAX_FILES_PER_RUN: 10`が定義されているが、実装には**ファイル数の上限制御がない**。フォルダに100枚の画像がある場合、すべて処理しようとしてGAS実行時間制限を超過する。

**改善案**: カウンタと上限チェックを追加する。残りファイルがある場合は次回トリガーに委ねる。

### 3.6 【中】HEIC→JPEG変換が疑似的

```js
const apiMimeType = (mimeType === 'image/heic' || mimeType === 'image/heif') ? 'image/jpeg' : mimeType;
```

MIMEタイプのラベルを`image/jpeg`に変えているだけで、**実際のバイナリ変換は行っていない**。HEICのバイナリをそのまま送信し、MIMEタイプだけJPEGと偽称している。Gemini APIがHEICバイナリをそのまま受け付ける場合は動作するが、MIMEタイプと実データの不一致はAPI挙動変更で壊れるリスクがある。

**改善案**: Gemini APIの公式ドキュメントでHEICサポート状況を確認し、対応していれば`image/heic`のまま送信する。非対応であればDrive APIのサムネイル機能等で実変換する。

### 3.7 【中】抽出失敗時のスプレッドシート記録がない

```js
if (extractedList && extractedList.length > 0) {
  // スプレッドシートに登録
} else {
  console.log(`抽出結果が空のためスキップ: ${file.getName()}`);
}
```

抽出結果が空の場合、ログに出力されるだけでスプレッドシートには何も残らない。ユーザーから見ると「アップロードした名刺が消えた」ように見え、追跡が困難。かつ、このファイルも入力フォルダに残り続けるため、3.2の無限再処理問題と同根。

**改善案**: 抽出失敗時も「処理日時・ファイル名・画像URL・ステータス（抽出失敗）」をスプレッドシートに記録し、ファイルはエラーフォルダに移動する。

### 3.8 【中】`processBusinessCards` 関数が長すぎる

`processBusinessCards`は実質100行超の関数で、以下の責務を1関数に詰め込んでいる:

1. フォルダからファイル取得
1. MIMEタイプのフィルタリング
1. Base64変換
1. HEIC MIMEタイプ変換
1. Gemini API呼び出し
1. スプレッドシート書き込み（ヘッダー生成含む）
1. ファイルリネーム
1. ファイル移動
1. ログ出力

**改善案**: 少なくとも「ファイルリネーム＋移動」と「スプレッドシート書き込み」を別関数に切り出す。

### 3.9 【低】ヘッダー行の自動生成ロジックの脆弱性

```js
if (sheet.getLastRow() === 0) {
  sheet.appendRow(["処理日時", "会社名", ...]);
}
```

ユーザーが誤ってヘッダー行を削除した場合、データ行がヘッダーなしで追加され続ける。また、ヘッダーの列順とデータ行の列順が文字列リテラルの並びでのみ対応しており、列を追加・変更する際にバグを生みやすい。

**改善案**: ヘッダー定義を定数配列として切り出し、データ行の構築もその定数のキーを参照するようにする。

### 3.10 【低】CLAUDE.md と README のモデル名不整合

CLAUDE.mdでは`gemini-1.5-flash`、README/Code.gsでは`gemini-2.5-flash-lite`。

### 3.11 【低】README に削除済みの経費証憑ドキュメントへの参照が残っている

ディレクトリ構成や「経費証憑 OCR ツール（別用途展開）」セクションに、既に削除された`経費証憑OCRツール_要件定義.md`と`経費証憑OCRツール_設計仕様書.md`へのリンクが残っている。リンク切れになるため、READMEからも削除が必要。

### 3.12 【低】ライセンス・リポジトリメタデータの未設定

LICENSE未定義、GitHubのDescription・Topicsが未設定。
