## エグゼクティブサマリー

コード単体の品質は及第点だが、**設計ドキュメント群と実際の実装の間に複数の根本的な乖離**がある。最も深刻なのは「外部API不使用・追加費用ゼロ」という設計方針と、Gemini API（外部有料API）を使用する実装が完全に矛盾している点だ。この乖離はプロジェクト全体の合意形成・方針の一貫性に対する信頼を損なう。個別のコード実装に目を向けると、タイムアウト対策・重複処理・エラー時のファイル退避など、設計書で明示的に約束していた安全装置が実装されていない。

---

## 1. 設計書と実装の整合性（Critical）

### 1-1. OCR方式の根本的な矛盾 🔴 Critical

これがプロジェクト全体で最大の問題。

**設計書（設計仕様書 §2.2 方針）の記述：**
> 外部サービス・有料APIは使用しない（Google Workspace標準機能のみ）

**設計書（実装プラン §1 ゴール）の記述：**
> 追加費用ゼロ（外部API不使用）でのシステム稼働

**実際の実装（Code.gs L116）：**
```javascript
const url = `https://generativelanguage.googleapis.com/v1beta/models/gemini-1.5-flash:generateContent?key=${apiKey}`;
```

実装はGemini APIという外部有料APIを使用している。設計書が採用を想定していたのは `Drive.Files.copy` によるGoogle Drive内蔵OCRだった（設計仕様書 §5.3.1）。

OCRエンジンの選択は「追加費用が発生するか」「個人情報（名刺データ）が外部サービスに送信されるか」という観点から、ステークホルダーに対して明示的に合意を取り直す必要がある事項だ。設計変更として文書化されないまま実装が先行している状態は、後から重大な問題に発展しうる。

**推奨対応：** 実装方針の変更をステークホルダーに明示し、設計仕様書を「Gemini API採用版」に改訂するか、Drive内蔵OCRに実装を戻すかを意思決定する。

---

### 1-2. フォルダ構成の不一致 🔴 Critical

**設計仕様書（§3.1）：** 4フォルダ構成
```
名刺管理/
├── 01_未処理/
├── 02_OCR中/         ← 処理中ステータス管理用
├── 03_処理済み/
└── 04_要確認/        ← OCR信頼度低 or 異常ファイル退避先
```

**実装（Code.gs CONFIG）：** 2フォルダ構成
```javascript
FOLDER_ID_INPUT: "...",      // = 01_名刺スキャン用
FOLDER_ID_PROCESSED: "...",  // = 02_処理済み名刺
```

`02_OCR中`（処理中ロック）と `04_要確認`（エラー・低信頼度退避）の2フォルダが完全に消えている。この結果、次の問題が連鎖して生じている。

- GASが並列トリガー実行された場合に同一ファイルを二重処理するリスクへの対策がない
- OCRが失敗・低信頼度だったファイルの退避先がなく、エラー時はファイルが元フォルダに放置される（後述）
- セキュリティ設計書で定義された「要確認」ステータスのワークフローが機能しない

---

### 1-3. スプレッドシートスキーマの不一致 🟠 High

**設計仕様書（§4.2）：** 12列（A〜L）
- OCRステータス列（I列）：「処理済み」「要確認」を色分け管理
- Webサイト列（H列）
- 備考列（K列）
- OCR生テキスト列（L列）：デバッグ・修正参考用
- 操作ログシート：GAS実行履歴記録

**実装（Code.gs L34）：** 8列のみ
```javascript
sheet.appendRow(["処理日時", "会社名", "部署・役職", "氏名", "メールアドレス", "電話番号", "住所", "画像URL"]);
```

設計書で運用フローの核となるOCRステータス管理（色分けワークフロー）が実装されていない。また操作ログシートも未実装のため、GASの実行履歴・エラー追跡ができない状態になっている。

---

### 1-4. 関数構成の不一致 🟡 Medium

**設計仕様書（§5.1）で定義された関数群：**

| 関数名 | 役割 |
|--------|------|
| `main()` | エントリーポイント |
| `ocrImage(fileId)` | OCR実行 |
| `parseBusinessCard(text)` | テキスト構造化 |
| `writeToSheet(data)` | スプレッドシート書き込み |
| `moveFile(fileId, toFolderId)` | ファイル移動 |
| `writeLog(msg, type)` | ログ記録 |

**実装の実態：**
- `processBusinessCards()` ── 上記6関数の全ロールを1関数に集約
- `extractWithGemini()` ── Gemini API呼び出しのみ

設計ドキュメントを参照して開発・保守を行おうとする人間にとって、ドキュメントと実装の対応が一切取れない状態。

## 2. コードレベルの問題

### 2-1. タイムアウト安全装置の欠如 🔴 Critical

実装プランとセキュリティ設計書の両方で明確に対策が必要と言及されていながら、実装されていない。

**実装プラン（§4. リスクと対策）の記述：**
> スクリプト側で「実行開始から5分経過したらループを中断する」安全装置を設け、残りの未処理ファイルは次回のトリガー実行時に回す仕様にする

**実際のコード（Code.gs L47〜98）：**
```javascript
while (files.hasNext()) {  // 終了条件：ファイルがなくなるまで無制限にループ
  const file = files.next();
  ...
}
```

GASの実行時間上限は6分。大量アップロード時（20枚超）はトリガー強制終了となり、処理中のファイルが破損状態（名前変更済みだが移動未完了など）になりうる。

---

### 2-2. エラー時のファイル放置問題 🟠 High

```javascript
try {
  // ...OCR処理...
  file.moveTo(processedFolder);  // 成功時のみ移動
} catch (e) {
  console.error(`エラー発生ファイル: ${file.getName()}, エラー詳細: ${e.message}`);
  // ← ここで何もしない。ファイルは入力フォルダに残る
}
```

エラーが発生したファイルが入力フォルダに残るため、次回のトリガー実行で再度処理が試みられる。一時的なAPIエラー（レートリミット等）なら再試行は正しい挙動だが、恒久的なエラー（壊れた画像ファイルなど）は永遠にループし続ける。エラー種別によって「要確認フォルダへ移動」と「リトライ」を分けるべき。

---

### 2-3. PropertiesServiceの非効率な呼び出し 🟡 Medium

```javascript
function extractWithGemini(base64Image, mimeType) {
  const properties = PropertiesService.getScriptProperties();
  const apiKey = properties.getProperty('GEMINI_API_KEY');  // 毎回呼び出し
  ...
}
```

`extractWithGemini`は1ファイルごとに呼ばれるため、10枚処理すると10回PropertiesServiceにアクセスする。GASのPropertiesServiceアクセスはネットワーク遅延を伴う。APIキーは`processBusinessCards()`の冒頭で一度だけ取得してから引数として渡すべき。これはまた「APIキーが設定されていない」エラーの検出タイミングをループ開始前に早めるという効果もある。

---

### 2-4. HEIC対応の不完全さ 🟡 Medium

```javascript
const apiMimeType = (mimeType === 'image/heic') ? 'image/jpeg' : mimeType;
```

2点の問題がある。まず、HEICのMIMEタイプには `'image/heif'` も存在するが未対応。次に、GeminiへHEICを `image/jpeg` として送信するのはMIMEタイプの偽称であり、Gemini APIがバイナリ構造を正しく読めるかは保証されていない。HEIC変換はGAS内での変換が難しいため、設計書の記述通りユーザーにJPEG設定を促すか、HEICを検知して `04_要確認` フォルダへ退避させる処理が正しい対応。

---

### 2-5. 重複処理防止機構の欠如 🟡 Medium

同一の名刺画像が複数回スプレッドシートに追記される経路が存在する。

- GASのトリガーが複数並列実行された場合
- `extractedData`取得後・`moveTo()`実行前に例外が発生した場合（appendRowは成功しているが移動が未完了になる）

ファイルIDをスプレッドシートに記録しておき、処理前にチェックするか、`02_OCR中`フォルダへの移動をロック機構として使う（設計書の4フォルダ構成の意図）べき。

---

### 2-6. ヘッダー行の毎回チェック 🟢 Low

```javascript
if (sheet.getLastRow() === 0) {
  sheet.appendRow([...]);
}
```

毎回の実行でシートの行数を確認するAPIコールが発生する。影響は軽微だが、初期化処理（ヘッダー・フォーマット設定）は`setup()`のような独立した関数として切り出し、初回セットアップ時に一度だけ実行する設計のほうが責務が明確で誤操作のリスクも下がる。

---

## 3. CLAUDE.md（開発規約）の矛盾

`src/CLAUDE.md` に示されているCONFIGのサンプルに問題がある。

**CLAUDE.mdのサンプル：**
```javascript
const CONFIG = {
  FOLDER_ID_INPUT:     "...",
  FOLDER_ID_PROCESSED: "...",
  SPREADSHEET_ID:      "...",
  SHEET_NAME:          "シート1",
  GEMINI_API_KEY:      "ここに取得したGemini APIキーを貼り付け"  // ← ハードコーディングを示唆
};
```

**実際のCode.gsの実装：**
```javascript
// スクリプトプロパティからAPIキーを安全に取得
const apiKey = properties.getProperty('GEMINI_API_KEY');
```

CLAUDE.mdはAPIキーをCONFIGオブジェクトに書く形式を示しているが、実装はPropertiesServiceから取得する形式になっており、セキュリティ設計書の方針とも一致している。CLAUDE.mdのサンプルが古いバージョンから更新されていない可能性が高く、このファイルを参照する開発者がAPIキーをコードに直書きするリスクを生む。

また `src/CLAUDE.md` の `## GAS 固有制約` でクラス構文を避けることを指示しているが、それ以上の構造化（例：ヘルパー関数への分割）に関する指針がなく、現在の「1関数モノリス」状態を正当化するかのような読み方もできる。意図を明確にすべき。

## 4. アーキテクチャ改善案

### 4-1. ドキュメントと実装の一致を最優先に回復する

現状の最大リスクは技術的負債ではなくドキュメントと実装の乖離だ。以下を優先順位順に対処する。

1. **OCR方式についての意思決定を文書化する** ── Drive内蔵OCRかGemini APIか、コストと品質のトレードオフを関係者と合意し、設計書に記録する。
2. **設計仕様書のフォルダ構成・スキーマを実装に合わせて改訂するか、実装を設計に合わせるかを決定する。**
3. **CLAUDE.mdのCONFIGサンプルを実際の実装と一致させる。**

---

### 4-2. 関数分割による保守性の向上

現在の`processBusinessCards()`はオーケストレーター・ファイル操作・スプレッドシート書き込み・エラーハンドリングを1関数に詰め込んでいる。GASはクラスを避けるべきだが、関数への分割は可能で推奨される。

提案する関数構成：

```
processBusinessCards()     ← エントリーポイント・ループ制御のみ
  ├── initSheet(sheet)     ← ヘッダー初期化（初回のみ）
  ├── processOneFile(file, sheet, processedFolder)  ← 1ファイルの処理
  │     ├── extractWithGemini(base64, mimeType)     ← API呼び出し（現行維持）
  │     └── appendCardRow(sheet, data, file)        ← シート書き込み
  └── writeLog(sheet, msg, level)                   ← ログシート記録
```

---

### 4-3. タイムアウト制御の実装パターン

```javascript
// processBusinessCards()の冒頭で開始時刻を記録
const startTime = new Date().getTime();
const TIMEOUT_MS = 5 * 60 * 1000; // 5分

while (files.hasNext()) {
  if (new Date().getTime() - startTime > TIMEOUT_MS) {
    console.log("タイムアウト安全装置発動。残りのファイルは次回トリガーで処理します。");
    break;
  }
  // ...処理...
}
```

---

### 4-4. エラー時のファイル退避パターン

```javascript
try {
  // ...処理...
} catch (e) {
  console.error(`エラー: ${file.getName()}, ${e.message}`);
  // エラー時は要確認フォルダへ退避（永続的なリトライループを防ぐ）
  if (reviewFolder) {
    file.moveTo(reviewFolder);
  }
}
```

エラーの種類（一時的なAPIエラーか恒久的な問題か）によってリトライ戦略を変えることも将来的には検討する。

## 5. ポジティブな評価点

批判的な指摘が中心になったが、以下は適切な実装として評価できる。

- **APIキーのPropertiesService管理** ── セキュリティ設計書の方針が正しく実装されている。
- **`muteHttpExceptions: true` とレスポンスコード検証** ── GASのエラーハンドリングとして正しいパターン。
- **`responseMimeType: "application/json"` の指定** ── JSONパース失敗を大幅に減らす適切な実装。
- **エラーログへのAPIキー・生レスポンスを含めない配慮** ── セキュリティ要件を満たしている。
- **ループ内のtry-catchによる1件失敗時の継続処理** ── バッチ処理として正しい設計。
- **`gemini-1.5-flash`の選択** ── 名刺OCRのユースケースに対してコストパフォーマンスが高いモデル選択。
